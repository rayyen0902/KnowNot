# AI 护肤助理系统工作流设计说明

> 版本：v1.0  
> 更新时间：2026-04-25  
> 说明：本文档定义用户从进入小程序到生成护肤报告、护肤日程、产品建议的完整业务工作流，帮助前后端和 AI 层保持一致。

---

## 一、工作流设计目标

### 1.1 让用户流程尽量自然
用户在小程序中的体验应尽量接近“被专业护肤顾问陪伴”的感觉，而不是机械填表。

### 1.2 让 AI 流程可控
AI 可以参与生成内容，但关键状态转换要由系统控制，不能完全依赖模型自由发挥。

### 1.3 让结果可追踪
每一步都要有记录，保证：

- 可回放
- 可断点续聊
- 可排查错误
- 可继续迭代优化

### 1.4 让未来扩展更容易
工作流设计要兼容：

- C 端用户体验
- B 端算法 API
- 私有模型接入
- 多轮报告迭代

---

## 二、整体流程总览

```text
用户进入小程序
→ 初始化身份
→ 授权/注册/登录
→ 收集皮肤资料
→ 上传照片/产品
→ AI 分析
→ 生成护肤报告
→ 生成护肤日程
→ 展示建议
→ 持续对话与迭代
```

---

## 三、用户采集流程

### 3.1 流程目标
采集用户的基础皮肤信息和当前护肤习惯，为后续分析提供输入。

### 3.2 采集内容
建议分批采集，而不是一次性填完：

- 肤质
- 年龄段
- 清洁习惯
- 化妆习惯
- 护肤习惯
- 过敏史
- 面部素颜照片
- 当前在用产品

### 3.3 建议交互方式
- 一问一答
- 支持跳过
- 支持稍后补充
- 支持用户返回修改

### 3.4 流程状态
```text
welcome
→ step_skin
→ step_age
→ step_habit
→ step_allergy
→ step_photo
→ collecting
```

### 3.5 说明
采集阶段不要过度打扰用户，建议优先收集最关键字段，再根据模型判断补充提问。

---

## 四、对话工作流

### 4.1 流程目标
通过自然对话方式完成信息采集，同时给用户提供即时护肤指导。

### 4.2 对话行为
用户可能会：

- 主动提问
- 简短回答
- 跳过问题
- 上传图片
- 表达抗拒或犹豫
- 纠正先前答案

系统需要支持这些场景。

### 4.3 对话处理逻辑

```text
用户消息进入
→ 读取当前 dialogue_state
→ 判断当前阶段
→ 提取关键信息
→ 更新 collected_fields
→ 计算配合度/情绪分
→ 决定是否继续追问
→ 返回回复
```

### 4.4 对话策略建议

| 用户状态 | 系统策略 |
|------|------|
| 配合回答 | 继续推进下一问题 |
| 回答含糊 | 追问一个更简单的问题 |
| 明显抗拒 | 降低问题密度，先安抚再继续 |
| 主动提问 | 先回答，再引导回流程 |
| 提供新信息 | 更新档案并重新判断分析条件 |

### 4.5 状态流转建议
```text
welcome
→ step_skin
→ step_age
→ step_habit
→ step_allergy
→ step_photo
→ collecting
→ analyzing
→ done
→ consulting
```

### 4.6 说明
对话流程不要求严格线性，允许用户打断、跳转和补充。系统只要能最终收齐关键字段即可。

---

## 五、AI 分析工作流

### 5.1 流程目标
把用户原始输入转成结构化的护肤分析结果。

### 5.2 输入数据
AI 分析一般会接收：

- 用户档案
- 当前护肤习惯
- 面部照片
- 当前使用产品
- 对话上下文
- AI 宪法 / Prompt 规则

### 5.3 分析步骤

```text
采集完成
→ 创建 ai_task
→ Python 拉取任务
→ 数据预处理
→ 拼接 Prompt / 规则
→ 调用模型或算法
→ 解析输出
→ 结构化结果
→ 写入报告表
→ 更新任务状态
```

### 5.4 输出建议
AI 输出最好包含：

- 肤质判断
- 主要问题
- 推荐成分
- 禁忌成分
- 产品搭配问题
- 早晚护肤建议
- 饮食建议
- 季节建议
- 置信度

### 5.5 任务状态
```text
pending
→ running
→ done
```

失败则进入：

```text
failed
```

### 5.6 异常处理
如果 AI 分析失败：

- 不要直接丢给用户报错
- 给用户返回友好提示
- 允许稍后重试
- 保留任务日志和输入参数

---

## 六、护肤报告工作流

### 6.1 流程目标
把 AI 分析结果整理成用户可以直接理解和使用的报告。

### 6.2 报告生成链路

```text
AI 结果返回
→ Go 侧校验
→ 去重与清洗
→ 结构化整理
→ 生成报告内容
→ 写入 skin_reports
→ 前端展示
```

### 6.3 报告内容建议
- 报告标题
- 总体评分
- 当前皮肤状态
- 问题清单
- 推荐成分
- 禁忌成分
- 当前产品搭配评价
- 饮食建议
- 季节建议
- 风险提示

### 6.4 报告版本
如果用户后续新增信息，或者重新分析，应该允许生成新版本报告。

建议：
- 旧报告保留
- 新报告新增 version
- 前端默认展示最新版本

### 6.5 说明
报告不是简单的文本输出，而是一个产品化内容，需要统一格式、术语和结构。

---

## 七、产品识别工作流

### 7.1 流程目标
识别用户正在使用的护肤产品，并判断是否适合当前肤质。

### 7.2 来源场景
产品数据可能来自：

- 用户手动输入
- 图片 OCR
- 订单同步
- 系统产品库匹配

### 7.3 识别流程

```text
用户上传产品图片
→ 存储图片 URL
→ OCR 提取文本
→ 提取成分与名称
→ 匹配系统产品库
→ 若匹配成功则补全信息
→ 若不匹配则保留自定义产品
→ 写入 user_products
```

### 7.4 输出内容
- 产品名称
- 品牌
- 类别
- 成分列表
- 适配性建议
- 搭配风险

### 7.5 说明
产品识别不要求 100% 自动正确，允许“半自动 + 用户确认”的方式逐步完善。

---

## 八、护肤日程工作流

### 8.1 流程目标
根据报告和用户产品生成可执行的早晚护肤日程。

### 8.2 输入内容
- 用户档案
- 报告结果
- 当前产品
- 季节
- 天气
- 当前日期

### 8.3 生成流程

```text
用户已完成分析
→ 获取报告和产品信息
→ 读取季节/天气
→ Python 生成日程
→ 结构化输出
→ 写入 skincare_schedules
→ 前端展示
```

### 8.4 日程内容建议
- 早间步骤
- 晚间步骤
- 每步用法
- 注意事项
- 饮食提醒
- 天气提醒

### 8.5 说明
日程建议尽量短、清楚、可执行，避免太长导致用户看不完。

---

## 九、持续对话工作流

### 9.1 流程目标
报告生成后，用户仍然可以继续向 AI 提问，形成长期陪伴关系。

### 9.2 场景举例
- 我这款面霜还能继续用吗？
- 我今天能不能刷酸？
- 我脸上最近爆痘了怎么办？
- 这个季节要怎么调整护肤步骤？

### 9.3 处理方式
- 读取用户当前报告
- 结合最新对话上下文
- 调用 AI 解释和建议
- 必要时触发重新分析

### 9.4 说明
持续对话应该尽量与报告、日程、产品库联动，而不是每次都从零开始问。

---

## 十、订单同步工作流

### 10.1 流程目标
如果后续接商品销售或商城，订单完成后可自动同步用户产品。

### 10.2 流程链路

```text
订单支付完成
→ 回调进入 Go
→ 校验订单状态
→ 解析商品明细
→ 匹配系统产品或创建自定义产品
→ 写入 user_products
→ 必要时触发产品建议更新
```

### 10.3 说明
这部分建议在第二阶段落地，当前 MVP 可以先不做。

---

## 十一、错误与降级工作流

### 11.1 AI 服务失败
如果大模型、OCR 或算法调用失败：

- 保留请求记录
- 记录错误原因
- 返回用户友好提示
- 支持重试

### 11.2 数据库写入失败
- 关键任务不要只依赖一次写入
- 写入失败要可重试
- 必要时保留任务中间态

### 11.3 图片识别失败
- 提示用户重新上传
- 可允许手动输入产品信息
- 不阻断主流程

### 11.4 报告生成超时
- 任务异步化
- 前端轮询状态
- 或通过消息通知用户结果完成

---

## 十二、推荐的状态控制方式

### 12.1 状态来源
建议主要由 `dialogue_states.phase` 控制流程。

### 12.2 状态更新原则
- 由 Go 控制状态流转
- Python 只给出建议状态
- 最终是否切换由业务层决定

### 12.3 好处
- 状态清晰
- 不容易乱跳
- 便于调试和回放
- 更适合后期商业化

---

## 十三、最小可用工作流

如果只做第一版 MVP，建议先跑通下面这条最小链路：

```text
用户进入
→ 初始化匿名身份
→ 收集肤质信息
→ 收集照片
→ 发送消息给 AI
→ 生成护肤报告
→ 展示报告
```

在这条链路跑通之后，再逐步补：

- 产品识别
- 日程生成
- 持续对话
- 订单同步
- B 端能力

---

## 十四、工作流设计结论

这个系统的工作流核心不是“让 AI 自由发挥”，而是：

1. 先把用户资料收齐
2. 再把数据整理结构化
3. 再把结构化数据交给 AI
4. 再把 AI 结果整理成产品化内容
5. 再沉淀为可复用的报告、日程和产品建议

这样既能快速上线，又能为后续私有算法、B 端 API 和长期用户运营打下基础。
