# AI 护肤助理系统架构方案

> 版本：v2.0  
> 更新时间：2026-04-25  
> 架构目标：快速上线、稳定承接用户、支持后续算法产品化与 B 端扩展  
> 技术方向：`Taro + React + TypeScript` 前端，`Go` 业务层，`Python` AI 层，`Supabase` 数据层，`FC` 作为早期 AI 接入适配层

---

## 一、项目背景

本项目是一个面向 C 端用户的 AI 护肤助理产品，核心能力包括：

- 用户皮肤状态采集
- 护肤习惯与照片采集
- AI 护肤分析与建议生成
- 护肤报告输出
- 产品搭配建议
- 成分推荐与禁忌提示
- 护肤步骤与手法建议
- 后续扩展为 B 端算法 API 服务

当前前端小程序已基本完成开发，数据库计划使用 `Supabase`，大模型接入层已在 `FC` 上做了初步实现。下一步重点是：

1. 保持快速落地
2. 让架构具备足够扩展性
3. 为后续私有算法、B 端 API、知识库、图像能力预留空间

---

## 二、设计原则

### 2.1 快速落地优先
第一阶段不追求过度微服务化，而是优先保证：

- 用户流程完整
- 接口稳定
- AI 能力可接入
- 报告可生成
- 可快速迭代

### 2.2 前后端职责分离
- 前端负责交互、采集、展示
- Go 负责业务编排、鉴权、对外 API
- Python 负责 AI 推理、图像处理、算法能力
- Supabase 负责数据存储
- FC 负责早期模型接入和轻量编排

### 2.3 AI 能力结构化输出
所有 AI 返回结果必须尽量结构化，避免直接依赖纯自然语言输出。

原因：

- 便于数据落库
- 便于前端渲染
- 便于二次整理和检索
- 便于未来 B 端 API 产品化

### 2.4 面向未来产品化
系统从一开始就要考虑后续能力：

- 私有算法接入
- B 端 API 售卖
- 多模型切换
- 报告版本管理
- 用户历史画像沉淀
- 知识库增强

## 四、服务职责划分

### 4.1 小程序前端

技术栈：

- `Taro`
- `React`
- `TypeScript`

职责：

- 用户注册 / 授权 / 登录
- 表单采集
- 图片上传
- 护肤问答交互
- 报告展示
- 产品建议展示
- 日程展示
- 历史记录展示

前端重点：

- 交互体验
- 采集效率
- 页面响应速度
- 结构化展示 AI 结果

---

### 4.2 Go 业务服务

Go 作为系统的业务主干，负责对外 API、鉴权和业务编排。

职责：

- 用户身份管理
- Token 鉴权
- 用户档案管理
- 报告查询与落库
- 产品库管理
- 任务编排
- AI 服务调用中转
- 后续计费与权限控制预留

适合 Go 的原因：

- 高并发请求处理能力强
- 类型安全
- 适合业务稳定层
- 易于做 API 网关与服务编排

建议 Go 主要承担“稳定业务层”，不要把大模型逻辑直接塞进去。

---

### 4.3 Python AI 服务

Python 负责系统中的“变化快、算法重、AI 强依赖”的部分。

职责：

- 用户皮肤分析
- Prompt 编排
- 大模型结果解析
- 图像处理
- OCR 识别
- 护肤规则推理
- 私有算法能力预留
- 知识检索增强
- 模型结果结构化输出

适合 Python 的原因：

- 生态适合 AI
- 图像和 OCR 库丰富
- Prompt / Agent / RAG 开发快
- 后续接私有模型更顺手

---

### 4.4 FC 函数层

你目前已经有一个 `FC` 函数层，这一层建议继续保留，但定位要明确：

它不是最终核心业务层，而是：

- 大模型 API 接入适配层
- 轻量数据处理层
- 临时编排入口
- 调试和快速实验层

适合做：

- 模型供应商切换
- Prompt 拼装
- 请求参数标准化
- 轻量转发
- 简单前处理 / 后处理

不建议长期承担：

- 复杂业务状态
- 大量持久化逻辑
- 任务流转主链路
- B 端正式 API 主入口

---

### 4.5 Supabase 数据层

建议第一阶段直接使用 `Supabase`，包括：

- PostgreSQL
- Auth
- Storage

职责：

- 用户数据存储
- 皮肤画像数据
- 护肤报告
- 用户产品库
- 图片资源
- 日程记录
- 对话状态记录

适合原因：

- MVP 上手快
- 省基础设施成本
- 便于快速验证产品
- 后续迁移成本可接受

## 五、核心业务流程

### 5.1 用户采集流程

用户进入小程序后，系统引导用户完成采集：

1. 用户基础信息授权
2. 采集肤质信息
3. 采集护肤习惯
4. 采集过敏史
5. 采集当前使用产品
6. 上传素颜照片
7. 整理成结构化画像
8. 进入 AI 分析流程

---

### 5.2 AI 护肤分析流程

数据流如下：

```text
用户输入
→ 前端采集
→ Go 业务接口
→ Python 数据预处理
→ 拼接 AI 宪法 / Prompt
→ 调用大模型或算法服务
→ 结构化结果解析
→ 报告生成
→ 落库
→ 返回用户
```

输出内容包括：

- 当前皮肤状态判断
- 存在的问题
- 推荐成分
- 不适合成分
- 当前产品搭配问题
- 护肤步骤建议
- 护肤手法建议
- 必要时产品图鉴
- 风险提示

---

### 5.3 报告生成流程

报告不是单纯模型原文，而是经过整理后的产品化内容。

步骤：

1. 接收 AI 输出
2. 进行结果结构化解析
3. 过滤重复信息
4. 统一术语与格式
5. 根据前端模板组装内容
6. 写入数据库
7. 返回前端展示

---

### 5.4 产品图鉴与成分建议流程

当用户上传护肤产品信息时：

1. 图片 OCR 或文本识别
2. 提取品牌、名称、成分
3. 与产品库匹配
4. 判断是否适合当前肤质
5. 输出搭配建议和成分建议
6. 若无匹配，则进入人工/规则补充流程

---

## 六、AI 宪法设计原则

你提到的“AI 宪法”建议不要只是一个长 prompt，而是一个**可版本化、可维护、可拆分的规则系统**。

### 6.1 AI 宪法内容建议拆为三层

#### 1. 角色层
规定模型是谁、说话风格是什么、不能做什么。

示例：

- 你是专业护肤顾问
- 语气温和、客观、不过度营销
- 不做医疗诊断
- 不输出不安全建议

#### 2. 规则层
规定分析逻辑：

- 如何判断肤质
- 如何识别护肤冲突
- 如何处理矛盾输入
- 如何处理缺失信息
- 如何建议早晚流程

#### 3. 输出协议层
规定必须输出的结构：

- 肤质判断
- 问题列表
- 成分建议
- 禁忌成分
- 早晚护肤步骤
- 产品搭配建议
- 可信度 / 置信度

---

### 6.2 AI 输出必须结构化

建议统一输出 JSON Schema，避免后端难以处理。例如：

```json
{
  "skin_type": "敏感偏干",
  "main_issues": ["屏障受损", "泛红", "干燥"],
  "recommended_ingredients": ["神经酰胺", "泛醇", "角鲨烷"],
  "avoid_ingredients": ["高浓度酸类", "强清洁成分"],
  "morning_routine": [],
  "night_routine": [],
  "product_tips": [],
  "confidence": 0.82
}
```

## 八、推荐接口分层

### 8.1 对外接口
由 Go 暴露给小程序前端使用。

建议统一前缀：

```text
/api/v1/...
```

示例：

- `POST /api/v1/user/init`
- `POST /api/v1/user/profile`
- `POST /api/v1/skin/submit`
- `POST /api/v1/report/generate`
- `GET /api/v1/report/:id`
- `GET /api/v1/report/list`
- `POST /api/v1/product/upload`
- `GET /api/v1/product/list`

---

### 8.2 内部接口
由 Go 调用 Python 使用。

建议仅内网访问，并带内部鉴权。

示例：

- `POST /internal/skin/analyze`
- `POST /internal/prompt/render`
- `POST /internal/ocr/recognize`
- `POST /internal/report/compose`
- `POST /internal/schedule/generate`

## 十四、架构结论

本系统的最佳架构方向不是“纯函数式 AI 接入”，也不是“一开始就重微服务化”，而是：

> **以小程序前端为入口，以 Go 为业务主干，以 Python 为 AI 算法核心，以 Supabase 为数据底座，以 FC 为早期 AI 适配层，逐步演进为可对外售卖的护肤算法平台。**

这个方案的优点是：

- 能快速上线
- 好维护
- 方便迭代
- 能承接私有算法
- 能支持未来 B 端 API 商业化
