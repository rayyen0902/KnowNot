# 多会话 Agent 与职责边界（真人 Owner + Cursor Agents）

> 用途：向不同 Cursor 会话分派任务时，**只喊下方 Agent 名**，职责不串线。  
> 原则：**Owner（真人）** 拍板价值观与商业；**项目总控 (PMO)** 对齐文档、拆里程碑与验收；其余为 **Cursor 内的执行类 Agent**。

---

## 协作模型（本项目事实）

1. **唯一真人参与者为 Owner（你本人）**：宪章与价值观底线、商业取舍、对外承诺与最终验收；**不隐含**「另有外包团队或第二真人开发」。  
2. **除 Owner 以外的开发、联调、巡检、拆单与文档统筹**，均由 **Cursor 中独立开启的 Agent 会话** 承担（下表各 Agent）；每个会话在首条或置顶消息中 **自报角色**（如「本会话：FE」）即可对齐边界。  
3. **PMO（项目总控）** 同样由 **Cursor 会话** 扮演，**非真人岗位**；由 Owner **指定**哪一会话承担 PMO（可与 FE/BE 不同会话）。PMO 负责：读 `docs/` 与《宪章》、拆里程碑、写可复制粘贴的实施命令与验收清单、冲突升级到 Owner。  
4. **Owner 需要把规范、计划、工单落进仓库时**，向 **已指派为 PMO 的 Cursor 会话** 下达写档/改档任务即可；执行类 Agent **不替代** Owner 做价值观与商业裁决。

---

## 一、架构总览（建议：**交互架构 (UX)** 与 **前端实现 (FE)** 分会话；质量巡检可由 PMO 兼岗）

```text
                    ┌─────────────────────────┐
                    │ Owner（唯一真人）        │  价值观 / 合规 / 商业拍板与最终验收
                    └────────────┬────────────┘
                                 │
                    ┌────────────▼────────────┐
                    │ 项目总控 (PMO)          │  对齐 docs 与《宪章》、里程碑、粘贴指令、验收标准
                    └────────────┬────────────┘
     ┌──────┬──────┬──────┬──────┼──────┬──────┬──────┬──────┐
     ▼      ▼      ▼      ▼      ▼      ▼      ▼      ▼
    UX     FE     BE   AI Eng DevOps Integr.   QA    Fix
 交互专岗 UI落地 Go+库 Python  部署   联调   抽检  缺陷修补
```

- **交互架构 (UX)**：用户动线、页面跳转、加载/空态/错误态、轮询与超时、手势与禁用态等 **先在 UX 定稿**，再交给 **前端实现 (FE)** 只做 **UI 与代码落地、修补样式**；避免「FE 会话又长又兼交互」导致反复出 bug。  
- **前端实现 (FE)**：以 **UX 交付的交互说明**（或 PMO 代写的短 spec）为唯一依据做组件与样式；**不再默认包办**复杂交互设计。  
- **质量巡检 (QA)**：仍可 PMO 兼岗；独立会话时只做抽检，不写日常功能。  
- **缺陷修复 (Fix)**：聚焦已确认缺陷的 **最小修复与回归验证**，不借机做大范围重构；边界与方案不明时先问 **UX / PMO** 再改代码。

## 二、Agent 一览表（分派用）

| Agent | 典型承载 | 职责边界（做什么 / 不做什么） |
|------|-----------|-------------------------------|
| **Owner** | **唯一真人**（你本人） | **做**：价值观与宪章底线、对外承诺口径、商业取舍、平台授权、最终验收。**不做**：替你承担「价值观/商业」类裁决的冒名；**可**亲自写代码，但不必兼任 PMO 文档统筹（可用 PMO 会话分担）。 |
| **项目总控 (PMO)** | **Cursor**「PMO / 总控」会话（**非真人**；由 Owner 指派） | **做**：读 `docs/` 与《宪章》、定优先级、拆里程碑、写「可复制粘贴」实施命令、验收清单、将规范**落实进仓库**、冲突升级到 Owner。**不做**：替你点平台后台、替你保管密钥；**不替代** Owner 做商业/宪章拍板。 |
| **交互架构 (UX)** | **Cursor**「交互 / UX」专会话（**非真人**） | **做**：用户路径与信息架构、关键状态机（含轮询/超时/失败分支）、页面间跳转与参数约定、动效与禁用规则；输出给 FE 的 **一页交互说明**（步骤列表 + 状态表即可）。**不做**：视觉稿细调、大量组件代码（交 FE）；不擅自改医学/合规话术（须对照宪章）。 |
| **前端实现 (FE)** | **Cursor**「前端 / UI」会话（**非真人**） | **做**：按 **UX**（或 PMO 代写的交互 spec）做 Taro/React **UI 与实现**、组件与样式、与接口字段对齐的展示；交互有歧义时 **先问 UX 或 PMO**，再改代码。**不做**：从零包办复杂交互方案（应交 UX）；Go/Python 核心；擅自改价值观话术。 |
| **后端工程 (BE)** | **Cursor**「Go + Supabase」会话（**非真人**） | **做**：对外 API、鉴权、会话与报告任务、落库、编排调 Python、错误码与 trace。**不做**：大段 Prompt 工程（交给算法工程）。 |
| **算法工程 (AI Eng)** | **Cursor**「Python + AI」会话（**非真人**） | **做**：Qwen 调用、Prompt 编排、结构化输出、图像/OCR、与宪章对齐的安全策略。**不做**：对外暴露 API（经 BE）。 |
| **运维发布 (DevOps)** | **Cursor**「运维 / FC」会话（**非真人**） | **做**：FC/轻量服务器、域名 HTTPS、环境变量、发布脚本、日志与告警入口。**不做**：产品文案与医学边界裁决（交 Owner/PMO）。 |
| **联调验收 (Integration)** | **Cursor**「联调」会话（**非真人**） | **做**：逐接口 cURL/Apifox、字段对齐表、Go↔Python↔前端三方对照、失败分支复现。**不做**：替你决策产品方向。 |
| **质量巡检 (QA)** | **Cursor** 独立会话，或由 PMO 会话兼岗（**非真人**） | **做**：上线前宪章敏感点抽检、关键路径回归清单、隐私与告知一致性。**不做**：日常功能开发。 |
| **缺陷修复 (Fix)** | **Cursor**「修 bug / hotfix」会话（**非真人**） | **做**：按工单或复现路径做 **定向修补**、补充/执行相关回归用例。**不做**：未经确认的需求扩展；与宪章/医学表述冲突的「顺手改文案」（应交 Owner 或 PMO）。 |

## 三、任务分派口令（复制即用）

- `【Owner】请确认：……（A/B，价值观或商业）`  
- `【项目总控 PMO】请输出：下一阶段目标 + 给 UX/FE/BE 的粘贴指令 + 验收标准`  
- `【交互架构 UX】请给出：…… 的交互说明（状态表 + 跳转 + 边界）供 FE 实现`  
- `【前端实现 FE】按 UX 交互说明 + implementation-v1 / 页面映射表执行：……（以 UI 与修补为主）`  
- `【后端工程 BE】按 api.md + integration-v1 执行：……`  
- `【算法工程 AI Eng】按 architecture 第六章 + 宪章执行：……`  
- `【运维发布 DevOps】按 infra 约定执行：……`  
- `【联调验收 Integration】按 integration-v1 与 cURL 清单执行：……`  
- `【质量巡检 QA】按宪章 + implementation-v1 做上线前抽检：……`  
- `【缺陷修复 Fix】按复现步骤最小修复：……（范围外先升级 PMO/UX）`

## 四、与仓库文档的对应关系

| Agent | 首要依据文档 |
|------|----------------|
| Owner | `宪章-AI护肤助理.md` |
| 项目总控 (PMO) | `README.md`（本目录索引）+ `implementation-v1.md` + `docs/Agent/README.md` |
| 交互架构 (UX) | `workflow.md` + `implementation-v1.md`（前端状态机、超时、绑定链路）+ `页面名称映射表.md` |
| 前端实现 (FE) | `页面名称映射表.md`（根目录）+ 小程序 `apps/miniapp` + **UX 当次交付的交互说明**（可贴在 issue/文档片段） |
| 后端工程 (BE) | `api.md` + `database.md` |
| 算法工程 (AI Eng) | `architecture.md`（AI 层）+ `宪章-AI护肤助理.md` |
| 运维发布 (DevOps) | `architecture.md`（部署相关段落）+ 后续 `infra/`（若有） |
| 联调验收 (Integration) | `api.md` + `integration-v1.md` |
| 质量巡检 (QA) | `宪章-AI护肤助理.md` + `workflow.md` |
| 缺陷修复 (Fix) | Issue/复现记录 + 对应模块代码与测试；涉接口契约时同步 `integration-v1.md` |

**定稿与工单摘要**：经 Owner 认可、供各 Agent 执行的交互说明片段、里程碑摘要等，可由 PMO 放入 **`docs/Agent/`**（见 `docs/Agent/README.md`）。

## 五、你问的那位「刚才干活出 cURL 的」是哪个 Agent？

**叫「联调验收 (Integration)」。**  
以后凡接口字段、cURL、三方对齐、联调记录，都归 **【联调验收 Integration】** 会话。

